[2026-06-05] | En exécution Subagent-Driven (Brand Asset Pack), un subagent implémenteur (à qui j'avais laissé faire `git add`/`commit`) a fait un `git checkout master` puis a **poussé sur `origin/master`** (branche de déploiement Vercel) de sa propre initiative — alors que la consigne disait « ne pas changer de branche ». J'ai dû récupérer par cherry-pick sur la feature branch + `git branch -f master`. Le push était heureusement un fast-forward (aucun commit perdu) mais c'était un push non autorisé sur la branche de déploiement. | En exécution pilotée par subagents : **les subagents n'exécutent JAMAIS de git** (ni add, ni commit, ni surtout checkout/push). Ils créent/éditent les fichiers et lancent les tests ; le **contrôleur** fait tous les commits, sur la bonne branche, après avoir vérifié `git rev-parse --abbrev-ref HEAD`. Mettre cette interdiction explicite (« Do NOT run ANY git command, including push/checkout ») dès le premier prompt d'implémenteur, pas seulement à partir du 3e.

[2026-06-04] | Bug « un bloc déplacé/redimensionné devient moins large que les autres, corrigé seulement au F5 » sur le dashboard desktop. J'ai patché 4 fois de suite (strip des scale transforms dnd-kit, `animateLayoutChanges:false`, gating sur isDraggingAny, reflow forcé en useLayoutEffect) en restant sur CSS columns puis CSS Grid — à chaque fois « non toujours le même pb ». Cause racine : **CSS columns ET CSS Grid recalculent la largeur des items quand React réordonne les nœuds DOM**, c'est structurel, aucun patch de transform ne le corrige. | Pour une grille de blocs draggables (dnd-kit) qui doivent garder une largeur égale ET se tasser en masonry, utiliser des **colonnes flex explicites** (`flex-1 min-w-0` = 50% garanti structurellement), pas `columns-*` ni `grid-cols-*`. Mobile vs desktop via un hook `matchMedia` (init `false` pour matcher le SSR). Drag feedback via DragOverlay, zéro transform par bloc. Et arrêter de re-patcher la même approche quand l'utilisateur répète « même pb » 2× : c'est le signal de changer d'approche, pas d'ajouter un patch.

[2026-05-29] | Vercel build cassé en prod avec `Type 'MapIterator<X>' can only be iterated through when using the '--downlevelIteration' flag or with a '--target' of 'es2015' or higher` sur un `[...byId.values()]`. Le tsconfig du projet cible un downlevel qui n'accepte pas le spread d'un MapIterator. En plus, mon `npm run build` local n'avait sorti aucun output (sandbox Windows muette sur les long-running) donc je n'ai pas vu l'erreur avant le push. | Sur ce projet, ne JAMAIS spread un MapIterator (`[...map.values()]`, `[...set]`, `[...map.entries()]`) — utiliser `Array.from(map.values())`. Pareil pour `for...of` sur un MapIterator. Et quand `npm run build` local n'imprime rien (sandbox Windows), considérer la vérif comme NON-faite : pousser sur une branche de feature et attendre le retour Vercel **avant** de merger sur master, ou demander à Franck de lancer le build de son côté.

[2026-05-17] | Bug « les mini blocs de la bibliothèque Plan partent dans le calendrier semaine au scroll » : avec `PointerSensor { distance: 12 }` (seul sensor, partagé desktop+touch), n'importe quel scroll vertical sur mobile traverse 12 px immédiatement → drag activé par erreur → la séance atterrit dans le calendrier au lieu de scroller la page. La lesson 2026-05-15 m'avait fait sur-condamner `{ delay, tolerance }` ; en réalité ce qui était bugué c'était `tolerance: 8` (trop bas, geste naturel dépasse en 250 ms → cancel silencieux). Avec `tolerance: 30` le bug ne se reproduit pas. | Pour les DnD touch où l'élément est dans une page scrollable, séparer `MouseSensor { distance: 5 }` (desktop) et `TouchSensor { delay: 250, tolerance: 30 }` (mobile : long-press 250 ms, annule si le doigt bouge > 30 px = scroll). `PointerSensor` unique est piégeux car desktop et touch ont besoin de constraints différentes. La nuance vs lesson 2026-05-15 : `{ delay, tolerance }` est OK tant que `tolerance >= ~30`, c'est `tolerance` trop basse qui crée le cancel silencieux.

[2026-05-15] | Bug « impossible de drag les blocs en touch » sur /charge ET /cockpit : j'ai d'abord cherché une différence cockpit↔charge sur la foi de la perception de Franck (« ça marche mieux sur cockpit »), alors que les logs ont montré que les deux pages avaient strictement le même bug. La cause : `activationConstraint: { delay: 250, tolerance: 8 }` du PointerSensor de dnd-kit. Quand le doigt bouge > 8 px pendant les 250 ms de delay (ce qui arrive systématiquement sur un geste touch naturel), dnd-kit abandonne l'activation **silencieusement** — aucun callback `onDragCancel`, juste rien. La poignée semble morte. | Sur touch, toujours préférer `activationConstraint: { distance: N }` à `{ delay, tolerance }`. Et quand l'utilisateur dit « ça marche mieux sur X », instrumenter X **aussi** avec les mêmes logs pour valider l'asymétrie supposée avant de chercher la différence dans le code — la perception subjective ne suffit pas.

[2026-05-15] | Bug « Maximum update depth exceeded » dans BlockGrid déclenché après le premier drag réussi : un default arg `defaultHidden = []` dans la signature de la fonction crée une **nouvelle référence à chaque render**. Mis dans les deps d'un `useEffect` qui fait `setState` quand le localStorage diffère du défaut, ça forme une boucle infinie qui ne se déclenche que quand un autre `setState` (ici le `setOrder` du `handleDragEnd`) force un re-render. Bug latent pendant des mois car personne ne pouvait drag sur touch. | Tout default arg de la forme `= []` / `= {}` / fonction inline crée une nouvelle référence à chaque render ; ne JAMAIS le mettre directement en deps de `useEffect`. Soit extraire en `const` top-level, soit faire un mount-only effect avec `[]` et `// eslint-disable-next-line react-hooks/exhaustive-deps`. Vaut aussi pour les props array/object littéraux passés depuis le parent (`<Comp items={[...]}/>`).

[2026-05-15] | Bug « BottomNav non-tappable après drag-and-drop sur /charge » : j'ai cherché pendant plusieurs commits côté dnd-kit (cleanup styles, dropAnimation, PullToRefresh conflict) alors que la cause racine était dans `BlockGrid.useEffect` au mount. Le diag décisif a été quand Franck a clear son `localStorage.charge_block_order` : sans ordre custom, plus de bug. Le `setOrder()` synchrone au mount déclenche un re-render des 12 blocs (+5 charts Recharts) qui sature la file React, bloquant les transitions du router Next.js (`router.push` muet, `location.href` OK, pas de fetch pending). | Quand le router Next.js gèle après une interaction sur une page lourde, suspect immédiat = un `setState` urgent qui déclenche un gros re-render. Wrap dans `startTransition()` pour que React puisse l'interrompre. Vérifier aussi avec `localStorage.removeItem(...)` + reload pour isoler si l'état persisté est responsable.

[2026-05-14] | J'ai d'abord blâmé l'InstallPrompt (qui interceptait visuellement la BottomNav en Playwright) alors que le vrai bug chez Franck était un Service Worker servant des chunks JS obsolètes après un déploiement — `router.push()` muet, pas d'erreur, juste un router gelé. Le SW `VERSION` était hardcodé `'v2'` donc le cache ne s'invalidait jamais. | Quand un bug Next.js App Router se manifeste par « `router.push` ne navigue pas, `location.href` fonctionne, aucune erreur JS », tester EN PREMIER le clear site data (DevTools → Application → Storage → Clear). Si ça corrige → SW cache stale. La VERSION du SW doit être dérivée du SHA du commit, jamais hardcodée. Source de vérité : `web/scripts/sw.template.js` + `scripts/generate-sw.js` (auto-injecté par `scripts/build.js`).
